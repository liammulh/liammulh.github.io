When it comes to writing automated tests for your code, there's a lot of jargon you need to wrap your head around. In this post I get to grips with some of that jargon.
Published: Thursday, October 5, 2023
Tagged under: testing

Before we get into the difference between mocks and stubs, we need some context. If you want a TL;DR, jump here.

I needed to test a function that gets data from an external API today, so I was googling on best practices for doing so. Tests are supposed to be deterministic and hermetic.

A correctly-written test is deterministic if the test passes when the implementation of the system under test is correct (A), and the test fails when the implementation of the system under test is incorrect. If the implementation of the system under test is correct, and the test passes most of the time, but occasionally fails, it's not deterministic. For example, suppose you have a test for correct code that gets data from an external API. Suppose the HTTP request for the data fails x percent of the time. Then your test fails x percent of the time, despite the code being correct. This test is not deterministic.

Google's book on software engineering [1] admonishes us to make tests hermetic:

All tests should strive to be hermetic: a test should contain all of the information necessary to set up, execute, and tear down its environment. Tests should assume as little as possible about the outside environment, such as the order in which the tests are run. For example, they should not rely on a shared database. This constraint becomes more challenging with larger tests, but effort should still be made to ensure isolation.

Webster's 1913 (B) provides a definition of the word "hermetic" [2]:

Made perfectly close or air-tight by fusion, so that no gas or spirit can enter or escape; as, an hermetic seal.

The entry on "hermetic" directs us to a note in the entry for "hermetically" [3]:

A vessel or tube is hermetically sealed when it is closed completely against the passage of air or other fluid by fusing the extremity; -- sometimes less properly applied to any air-tight closure.

If we test code that reaches out to an external API, our test probably won't be deterministic because the request could fail. Our test isn't hermetic because it relies on an external dependency, the API. How, then, do we test our code? We use a test double. A test double is something that stands in for the real thing. In this case, we will use a test double for our API request. There are different types of test doubles, according to Martin Fowler [4]:

  • Dummy objects are passed around but never actually used. Usually they are just used to fill parameter lists.
  • Fake objects actually have working implementations, but usually take some shortcut which makes them not suitable for production (an in-memory database is a good example).
  • Stubs provide canned answers to calls made during the test, usually not responding at all to anything outside what's programmed in for the test.
  • Spies are stubs that also record some information based on how they were called. One form of this might be an email service that records how many messages it was sent.
  • Mocks are what we are talking about here: objects pre-programmed with expectations which form a specification of the calls they are expected to receive.

I will attempt to summarize my understanding of the difference between mocks and stubs based on my reading of [4]. Generally a stub is used for state testing, i.e. it tests whether a function returns the correct value. Generally a mock is used for behavior testing, i.e. whether a function took the correct steps. Mocking libraries like Python's unittest.mock provide utilities for testing whether the system under test took the correct steps.

Notes

  • (A) A test doesn't really test whether the implementation of the system under test is correct. It tests whether it passes the test. Whether the test actually tests what it needs to test is a matter of how well-written the test is. A test should capture the requirements of the system under test. go back
